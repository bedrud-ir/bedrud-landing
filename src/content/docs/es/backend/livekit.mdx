Bedrud utiliza **LiveKit** para manejar la comunicación de video y audio en tiempo real. LiveKit proporciona el servidor de medios SFU (Selective Forwarding Unit), y Bedrud maneja la autenticación, la gestión de salas y los controles de administrador.

## Modo Incrustado vs. Externo

Bedrud admite dos modos de despliegue de LiveKit:

1.  **Modo Incrustado (Predeterminado):** El backend inicia y gestiona un proceso del servidor LiveKit internamente. No se requiere infraestructura adicional: el backend maneja el ciclo de vida del proceso LiveKit.
2.  **Modo Externo:** Bedrud se conecta a un servidor o clúster de LiveKit separado. Esto es útil para escalabilidad horizontal o al usar una instancia administrada de LiveKit Cloud.

### Configuración del Modo Externo

Para usar un servidor LiveKit externo, establece las siguientes claves en `config.yaml`:

```yaml
livekit:
  host: "livekit.example.com:7880"
  api_key: "your-api-key"
  api_secret: "your-api-secret"
  embedded: false
```

Cuando `embedded` es `false`, Bedrud omite iniciar el binario LiveKit incrustado. La clave de API y el secreto deben coincidir con las credenciales del servidor externo.

Para la configuración y configuración del clúster de LiveKit, consulta la [documentación de LiveKit](https://docs.livekit.io).

## Cómo Funciona

### 1. Creación de Sala

Cuando un usuario crea una sala en Bedrud, el servidor no crea una sala de LiveKit inmediatamente. Las salas de LiveKit se crean bajo demanda cuando se une el primer participante.

### 2. Tokens de Unión

Cuando un usuario se une a una reunión:

1.  El frontend envía una solicitud a `/api/room/join`.
2.  El backend verifica que el usuario tiene permiso para unirse a esa sala.
3.  El backend usa su clave de API y secreto para generar un JWT firmado (Join Token).
4.  El token contiene:

    - El nombre de la sala.
    - La identidad del usuario (nombre para mostrar).
    - Permisos: por ejemplo, si el usuario puede publicar audio o compartir su pantalla.
5.  El frontend recibe este token y se conecta directamente al puerto de medios de LiveKit (predeterminado `7880`).

### 3. Controles de Sala (Admin)

El backend usa el Go SDK de LiveKit para realizar acciones administrativas:

- **Expulsar:** Desconecta a un participante.
- **Silenciar:** Silencia a la fuerza el micrófono de un participante.
- **Permisos:** Cambia lo que un participante puede hacer en tiempo real.

## Arquitectura de Red

- **Puerto API (8090/443):** Maneja solicitudes HTTP y señalización WebSocket para la configuración de llamadas.
- **Puerto de Medios (7880):** Maneja datos de video y audio usando protocolos WebRTC. La alternativa ICE/TCP usa el puerto 7881 cuando UDP está bloqueado.
- **Puerto TURN (3478 UDP / 5349 TLS):** Retransmite medios para clientes detrás de NAT restrictivos o firewalls. Consulta la [Guía del Servidor TURN](/es/docs/architecture/turn-server).

Para los requisitos de firewall y puertos, consulta [Conectividad WebRTC](/es/docs/architecture/webrtc-connectivity).

## Manejo de Errores

### LiveKit Inalcanzable

Si el servidor LiveKit es inalcanzable, la generación de tokens aún puede tener éxito (los tokens se crean localmente usando la clave de API y el secreto). Sin embargo, el cliente fallará al conectarse al servidor de medios. Síntomas:

- El cliente recibe un token de unión válido pero la conexión WebRTC se agota.
- El SDK de LiveKit emite un evento `Reconnecting` o un error de conexión.

**Verificar:** Verifica que el servidor LiveKit esté en ejecución (`systemctl status livekit`) y que el host/puerto en `config.yaml` sea correcto.

### Bloques del Proceso LiveKit Incrustado

En modo incrustado, si el proceso secundario de LiveKit falla:

- El servidor API de Bedrud continúa ejecutándose (se pueden listar salas, los usuarios pueden iniciar sesión).
- Las reuniones activas pierden la conectividad de medios.
- Las nuevas solicitudes de unión generarán tokens, pero los clientes no podrán conectarse.

**Verificar:** Inspecciona los registros en `/var/log/bedrud/bedrud.log` o ejecuta `journalctl -u livekit -f` para obtener detalles del bloqueo.

### Expiración de Tokens

Los tokens de unión de LiveKit tienen una ventana de validez corta. Si un cliente espera demasiado tiempo entre recibir el token y conectarse, el token expirará. El cliente debe solicitar un nuevo token a través de `/api/room/join`.
