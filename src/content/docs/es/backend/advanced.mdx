Este documento cubre las decisiones técnicas internas detrás de la arquitectura de binario único de Bedrud.

## 1. El Puente Fiber a Estándar
Bedrud utiliza **Fiber** por su velocidad, pero algunas dependencias como **Goth** (OAuth) y **LiveKit SDK** (Twirp) esperan `http.ResponseWriter` y `http.Request` estándar de Go.

Para conectar estos dos mundos, Bedrud implementa un **Patrón Adaptador** en `internal/handlers/auth.go`:

### Adaptador `responseWriter`
```go
type responseWriter struct {
    ctx     *fiber.Ctx
    headers http.Header
    status  int
}
```
Este struct implementa la interfaz `http.ResponseWriter`. Cuando una función de biblioteca estándar (como Goth) llama a `Header().Add()`, el adaptador lo almacena en un mapa `headers` local. Cuando se llama a `WriteHeader()`, copia manualmente todos los encabezados de vuelta al **Contexto de Fiber**.

### Propósito
Permite que Bedrud siga siendo extremadamente rápido (usando Fiber) mientras mantiene la compatibilidad completa con el vasto ecosistema de bibliotecas web estándar de Go.

## 2. Proxy Inverso de LiveKit
Bedrud evita problemas de gestión de puertos enrutando toda la señalización de medios a través de su puerto principal.

### Cómo funciona:

1.  **Detección:** Si `LIVEKIT_INTERNAL_HOST` apunta a `127.0.0.1`, Bedrud inicia el motor de medios interno.
2.  **Montaje:** Monta un `httputil.NewSingleHostReverseProxy` en la ruta `/livekit`.
3.  **Eliminación de Ruta:** Se usa una función **Director** personalizada para eliminar el prefijo `/livekit` para que el servidor de medios reciba la solicitud como si estuviera en la raíz.
4.  **Soporte de WebSocket:** Dado que el proxy opera en el flujo TCP subyacente a través de `adaptor.HTTPHandler`, admite de forma nativa los WebSockets usados por LiveKit para la señalización en tiempo real.

## 3. Eficiencia de Almacenamiento: Structs Incrustados
Bedrud hace uso intensivo de la etiqueta `embedded` de GORM para mantener el esquema de la base de datos plano y optimizado.

**Ejemplo de `models/room.go`:**
```go
type Room struct {
    ID       string       `gorm:"primaryKey"`
    Settings RoomSettings `gorm:"embedded;embeddedPrefix:settings_"`
}
```
Esto hace que GORM cree columnas como `settings_allow_chat` directamente en la tabla `rooms`. Esto es más rápido que un `JOIN` y más searchable que un blob `JSON`.

## 4. Infraestructura de Pruebas
Bedrud sigue una estrategia de pruebas "Primero la Base de Datos" usando el paquete `internal/testutil`.

### Lógica de `SetupTestDB`:

- Crea una base de datos **SQLite en memoria** única para cada prueba.
- Ejecuta automáticamente todas las migraciones.
- Proporciona una pizarra limpia, asegurando que las pruebas sean deterministas y puedan ejecutarse en paralelo.

Para ejecutar pruebas del backend:
```bash
go test ./internal/...
```

## 5. Seguridad: Rotación y Revocación de Tokens
A diferencia de las implementaciones simples de JWT, Bedrud rota **ambos** tokens en cada actualización.

- **Detección de Reutilización del Token de Actualización:** Una vez que se usa un token de actualización para obtener un nuevo par, se bloquea.
- **Limpieza:** La tabla `BlockedRefreshToken` garantiza que incluso si se roba un token de actualización, no puede usarse después de que el usuario haya cerrado sesión limpiamente o actualizado su sesión.
