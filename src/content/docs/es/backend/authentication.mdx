Bedrud admite múltiples métodos de autenticación.

## JWT (JSON Web Tokens)
Bedrud utiliza JWT para la gestión de sesiones. Al iniciar sesión, el servidor devuelve dos tokens:

1.  **Token de Acceso:** De corta duración (ej., 1 hora). Se utiliza para acceder a endpoints de API protegidos.
2.  **Token de Actualización:** De larga duración (ej., 30 días). Se utiliza para obtener un nuevo Token de Acceso sin volver a iniciar sesión.

El backend valida estos tokens en el archivo `internal/middleware/auth.go`.

## Gestión y Seguridad de Tokens

### 1. Estrategia de Par de Tokens (Rotación)
Bedrud utiliza una estrategia segura de rotación de tokens. Cuando un usuario se autentica, recibe:

- **Token de Acceso:** Firmado con HS256, contiene ID de usuario, nombre y permisos. Válido por una duración corta (configurado vía `tokenDuration`).
- **Token de Actualización:** Un token de larga duración separado.

**Rotación:** Cuando el Token de Acceso expira, el cliente envía el Token de Actualización a `/api/auth/refresh`. El servidor entonces emite un **nuevo par completo** de tokens y actualiza el token de actualización almacenado en la base de datos. Esto limita la ventana de oportunidad incluso si un token es robado.

### 2. Revocación de Tokens (Lista Negra)
Para admitir el cierre de sesión seguro, Bedrud implementa una **Lista Negra de Tokens**.

- Cuando un usuario cierra sesión, su Token de Actualización actual se agrega a la tabla `blocked_refresh_tokens`.
- Antes de actualizar cualquier sesión, el servidor verifica esta tabla. Si un token está bloqueado, la solicitud se rechaza inmediatamente.
- Una tarea en segundo plano (Scheduler) está diseñada para limpiar estos tokens bloqueados una vez que expiran naturalmente.

### 3. Estructura de Claims JWT
Los tokens llevan metadatos que permiten al backend y al frontend tomar decisiones rápidas sin consultas a la base de datos:
```json
{
  "userId": "uuid-string",
  "email": "user@example.com",
  "name": "Display Name",
  "accesses": ["user", "admin"],
  "exp": 123456789,
  "iat": 123456780
}
```

## Métodos de Autenticación

### 1. Correo Electrónico y Contraseña Local

- Las contraseñas se hashean usando **bcrypt** - nunca se almacenan como texto plano.
- Cuando te registras, se crea un nuevo registro de Usuario en la base de datos.

### 2. Inicio de Sesión Social (OAuth2)
Bedrud utiliza la biblioteca `Goth` para admitir:

- **Google**
- **GitHub**
- **Twitter**

Puedes habilitar estos llenando la sección `auth:` en tu `config.yaml` con tu ID de Cliente y Secreto de API.

### 3. Passkeys (WebAuthn/FIDO2)
Bedrud admite autenticación sin contraseña a través del estándar FIDO2 (WebAuthn).

- **Ceremonia de Registro:**
  1.  **Inicio:** El servidor genera un **Desafío** criptográficamente seguro y lo almacena en la sesión del usuario.
  2.  **Finalización:** El autenticador del cliente (ej., TouchID, YubiKey) firma el desafío. El servidor verifica la atestación, extrae la **Clave Pública** y la guarda en la tabla `passkeys`.
- **Ceremonia de Autenticación:**
  1.  **Inicio:** El servidor emite un nuevo desafío.
  2.  **Finalización:** El cliente firma el desafío con su clave privada. El servidor recupera la Clave Pública de la base de datos, verifica la firma y verifica el **Contador de Firma**.
- **Seguridad:** La coincidencia del campo `Counter` previene **Ataques de Reproducción**. Si el contador del autenticador no es más alto que el contador almacenado, el inicio de sesión se rechaza.

### 4. Inicio de Sesión de Invitado
Permite a los usuarios unirse a una reunión instantáneamente con solo un nombre. El servidor crea un registro de usuario temporal con el rol `guest`.

## Interoperabilidad y CORS
Bedrud está diseñado para ser accesible desde múltiples orígenes, lo cual es crucial para el desarrollo (servidor de desarrollo de React) y la producción (diferentes subdominios).

- **Política de CORS:** Definida en `config.yaml`. Admite `AllowCredentials: true`, que es necesario para las cookies de sesión utilizadas durante las ceremonias de OAuth y Passkey.
- **Orígenes Permitidos:** Por defecto, permite el dominio propio del servidor y `localhost:3000` (el puerto del servidor de desarrollo de React).

## Roles de Usuario (Control de Acceso)
Cada usuario tiene una lista de "accesses" (roles).

- `user`: Usuario estándar. Puede unirse y crear salas.
- `guest`: Usuario temporal. Puede unirse a salas.
- `superadmin`: Puede gestionar todos los usuarios y salas a través del panel `/admin`.

El guardia `middleware.RequireAccess("superadmin")` protege las rutas de administración.
