## Hoja de Ruta del Proyecto y Árbol

El directorio `server/` contiene el backend en Go y el servidor de medios LiveKit incrustado. El directorio `apps/` contiene las aplicaciones cliente.

```text
bedrud/
├── apps/                # Aplicaciones Cliente
│   ├── web/             # Frontend de React (TanStack Start)
│   ├── android/         # Aplicación Nativa de Android
│   └── ios/             # Aplicación Nativa de iOS
├── server/              # Backend en Go (El "Dispositivo")
│   ├── cmd/             # Puntos de entrada de CLI (run, install)
│   ├── internal/        # Código de aplicación privado
│   │   ├── auth/        # Lógica de JWT, Passkeys, OAuth
│   │   ├── handlers/    # Controladores HTTP
│   │   ├── models/      # Esquemas de Base de Datos
│   │   ├── repository/  # Capa de Acceso a Datos de GORM
│   │   └── livekit/     # Gestión del Servidor de Medios
│   ├── frontend/        # (Generado) Activos web compilados
│   ├── config.yaml      # Plantilla de configuración
│   └── Makefile         # Automatización de compilación y despliegue
└── docs/                # Documentación del Sistema (MkDocs)
```

## Mapa de Directorios

### `/cmd/bedrud`
- **`main.go`**: El punto de entrada de la aplicación. Maneja comandos de CLI como `run`, `install`, y `livekit`.

### `/config`
- **`config.go`**: Define la estructura `Config` y carga configuraciones desde `config.yaml` o Variables de Entorno.
- **`livekit.yaml`**: Configuración predeterminada para el servidor LiveKit incrustado.

### `/internal`

- **`/auth`**: contiene lógica para generación de JWT, proveedores de OAuth, y registro/inicio de sesión de Passkey (WebAuthn).
- **`/database`**: gestiona la conexión a SQLite o PostgreSQL y ejecuta migraciones.
- **`/handlers`**: Manejadores HTTP de Fiber. Aquí encuentras la lógica para cada endpoint de API (ej., `auth_handler.go`, `room_handler.go`).
- **`/models`**: Modelos de base de datos de GORM. Estos archivos definen las tablas en tu base de datos.
- **`/repository`**: La "Capa de Acceso a Datos". Los manejadores llaman a los repositorios en lugar de escribir consultas de base de datos directamente, manteniendo la lógica de los manejadores limpia.
- **`/livekit`**: Integración con el SDK de Go de LiveKit. Gestiona la creación de salas y genera "Tokens de Unirse" para los participantes.
- **`/middleware`**: Middleware personalizado de Fiber para verificaciones de autenticación, registro y CORS.
- **`/server`**: El código que une todo. Inicializa la base de datos, repositorios e inicia el servidor Fiber.
- **`/install`**: Lógica para el comando `bedrud install` que automatiza la configuración de systemd y TLS en Linux.
- **`/scheduler`**: Tareas en segundo plano (si las hay).
- **`/utils`**: Pequeñas funciones auxiliares (ej., hashing de contraseñas, cadenas aleatorias).

### `/migrations`

- contiene archivos SQL o Go para actualizaciones del esquema de base de datos.

### `/docs` (dentro de server)

- contiene archivos de documentación de Swagger/OpenAPI (generados por `swag`).

## Análisis Técnico Profundo

### 1. Incrustación de Binarios (El "Truco")

Bedrud utiliza la directiva `//go:embed` para agrupar archivos en el binario compilado.

- **Frontend:** La carpeta `dist/client/` de React (más un `index.html` pre-renderizado) está incrustada en `server/ui.go`. Los archivos estáticos se sirven directamente desde la memoria utilizando el middleware `filesystem` de Fiber.
- **Servidor LiveKit:** El ejecutable pre-compilado `livekit-server` está incrustado en `internal/livekit/bin/`. En tiempo de ejecución, Bedrud lo extrae a `/tmp/bedrud-livekit-server` y lo inicia como un proceso en segundo plano (`internal/livekit/server.go`).

### 2. Proxy Inverso de LiveKit

Para evitar abrir múltiples puertos (señalización, API, etc.), Bedrud enruta todo el tráfico de señalización de LiveKit a través de su puerto HTTP(S) principal.

- Cualquier solicitud que comience con `/livekit` es interceptada en `internal/server/server.go`.
- Un **Proxy Inverso** (usando `httputil.NewSingleHostReverseProxy`) reenvía estas solicitudes a la instancia interna de LiveKit (generalmente ejecutándose en `127.0.0.1:7880`).
- El proxy elimina el prefijo `/livekit` antes de reenviar, permitiendo que LiveKit funcione como si fuera la aplicación raíz.

### 3. Contexto y Variables Locales de Middleware
El backend utiliza `.Locals` de Fiber para pasar datos entre middleware y manejadores.
- **Middleware de Autenticación (`internal/middleware/auth.go`):** Valida el JWT y almacena el objeto `Claims` en `c.Locals("user")`.
- **Manejadores:** Pueden acceder al ID y permisos del usuario actual usando:
  ```go
  claims := c.Locals("user").(*auth.Claims)
  userID := claims.UserID
  ```

### 4. Sobrescritura de Configuración

Aunque Bedrud utiliza un archivo `config.yaml`, casi todas las configuraciones se pueden sobrescribir usando **Variables de Entorno**. Esto es esencial para entornos de Docker y CI/CD.

| Variable | Descripción |
|----------|-------------|
| `SERVER_PORT` | El puerto en el que escucha el backend (predeterminado: `8090`). |
| `SERVER_ENABLE_TLS` | Booleano (`true/false`) para habilitar HTTPS. |
| `SERVER_DOMAIN` | Tu dominio de producción (usado para ACME y RP ID de Passkey). |
| `DB_TYPE` | `sqlite` o `postgres`. |
| `DB_PATH` | Ruta al archivo `.db` (si se usa SQLite). |
| `LIVEKIT_HOST` | La URL pública para LiveKit (ej., `https://meet.example.com/livekit`). |
| `LIVEKIT_API_KEY` | Clave para autenticación de LiveKit. |
| `JWT_SECRET` | Clave secreta utilizada para firmar Tokens de Acceso. |

## Estándares de Código y Patrones

El backend sigue estos patrones:

### 1. Patrón de Repositorio
Los manejadores no deben comunicarse directamente con la base de datos. Usa un Repositorio. Esto hace que el código sea más fácil de probar y permite cambios en la lógica de la base de datos sin tocar los manejadores de API.

### 2. Manejo Estándarizado de Errores
Los manejadores de API deben devolver mensajes de error claros.
- Usa `c.Status(fiber.StatusBadRequest).JSON(...)` para errores de validación.
- Usa `c.Status(fiber.StatusUnauthorized).JSON(...)` para errores de autenticación.
- Usa `c.Status(fiber.StatusInternalServerError).JSON(...)` para errores de base de datos o del servidor.

### 3. Registro Estructurado
El backend utiliza **Zerolog** para el registro.
- `log.Info()`: Para eventos importantes (ej., servidor iniciado).
- `log.Error()`: Para fallas.
- `log.Debug()`: Para información detallada de desarrollo.

Evita usar `fmt.Println` para el registro en la lógica central.

### 4. Convenciones de Nomenclatura
- **Archivos:** snake_case (ej., `user_handler.go`).
- **Estructuras/Funciones:** PascalCase (ej., `GetUserByEmail`).
- **Variables:** camelCase (ej., `hashedPassword`).
