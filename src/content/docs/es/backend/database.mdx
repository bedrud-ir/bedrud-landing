Bedrud utiliza **GORM** como su ORM, soportando tanto SQLite como PostgreSQL.

## Bases de Datos Soportadas
- **SQLite:** Utilizado por defecto para desarrollo fácil e instalaciones pequeñas. La base de datos se almacena en un archivo (generalmente `bedrud.db`).
- **PostgreSQL:** Recomendado para entornos de producción con muchos usuarios.

Puedes configurar el tipo de base de datos y los detalles de conexión en el archivo `config.yaml` en la sección `database:`.

## Modelos Principales

Los modelos se encuentran en `internal/models/`. Aquí están los más importantes:

### Usuario (`user.go`)

Almacena información de la cuenta incluyendo credenciales y roles.

### Sala (`room.go`)
Representa una sesión de reunión. Utiliza **Estructuras Incrustadas** de GORM para organizar configuraciones.
En lugar de una tabla separada o un blob JSON, los campos de `RoomSettings` (como `allow_chat`, `allow_video`) se almacenan directamente en la tabla `rooms` con un prefijo `settings_` (ej., `settings_allow_chat`). Esto proporciona un modelo limpio en Go mientras mantiene una estructura de base de datos plana y de alto rendimiento.

### Passkey (`passkey.go`)

Almacena claves públicas FIDO2/WebAuthn para inicio de sesión sin contraseña.

### Tipos Personalizados: `StringArray`

Las bases de datos SQL estándar manejan matrices de manera diferente (PostgreSQL tiene matrices nativas, SQLite no). Para mantener la compatibilidad, Bedrud define un tipo `StringArray` en `internal/models/user.go`.

- Implementa `sql.Scanner` y `driver.Valuer`.
- En **PostgreSQL**, utiliza el tipo nativo `text[]`.
- En **SQLite**, serializa la matriz en una cadena (ej., `{admin,user}`) para el almacenamiento.

### Gestión de Claves Foráneas

El `AutoMigrate` de GORM no maneja claves primarias compuestas (ej., `room_participants`).

En `internal/database/migrations.go`, Bedrud ejecuta manualmente sentencias `ALTER TABLE` para asegurar que las restricciones de clave foránea (como `ON DELETE CASCADE`) se apliquen correctamente en producción (PostgreSQL).

## Patrón de Repositorio

El backend utiliza el **Patrón de Repositorio** para el acceso a la base de datos. Los manejadores no llaman a GORM directamente - usan un repositorio.

Ejemplo:
```go
// En el manejador:
user, err := h.userRepo.GetByEmail(email)

// En el repositorio:
func (r *UserRepository) GetByEmail(email string) (*models.User, error) {
    var user models.User
    err := r.db.Where("email = ?", email).First(&user).Error
    return &user, err
}
```

## Migraciones Automáticas

Cuando el servidor se inicia, ejecuta automáticamente "AutoMigrate". Esto crea o actualiza las tablas de la base de datos basándose en las estructuras de Go en `internal/models`. No se requieren sentencias manuales `CREATE TABLE` para cambios simples de esquema.
