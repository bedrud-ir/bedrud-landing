Esta guía cubre la instalación del **servidor Bedrud** (API, interfaz web, servidor de medios WebRTC) en un servidor Linux. Para aplicaciones cliente de escritorio y móvil, consulta [Instalación del Cliente](/es/docs/getting-started/clients).

---

## Requisitos Previos

| Requisito | Detalles |
|-----------|---------|
| **Sistema operativo** | Debian 12+, Ubuntu 22.04+, Arch Linux (rolling), Fedora 39+ |
| **Arquitectura** | amd64 (x86_64) o arm64 (aarch64) |
| **Acceso** | Root o sudo |
| **Sistema de inicio** | systemd |
| **Hardware** | 1 vCPU, 512 MB de RAM mínimo. 2 vCPU, 2 GB de RAM recomendados para 10+ salas concurrentes |
| **Puertos** | `8090` (API + interfaz web), `7880` (LiveKit WebSocket), `50000–60000/udp` (medios WebRTC) |
| **TLS automático** | Adicionalmente abre `80` y `443`. Requiere un dominio con registro DNS A que apunte a este servidor |
| **Herramientas** | `curl`, `tar` (preinstalado en la mayoría de distribuciones) |

> **¿Sin dominio?** Bedrud funciona con una dirección IP usando TLS autofirmado. Los navegadores mostrarán una advertencia de certificado pero el video y el audio funcionan. Para certificados de confianza en redes internas, consulta la [Guía de TLS Interno](/es/docs/guides/internal-tls).

## 2. Docker

Ejecuta Bedrud en un contenedor: sin configuración de servidor Linux, sin systemd, sin descarga de binario.

### Imagen Precompilada (la más rápida)

```bash
docker pull ghcr.io/bedrud-ir/bedrud:latest

docker run -d --name bedrud \
  -p 8090:8090 \
  -p 7880:7880 \
  -p 50000-60000:50000-60000/udp \
  -v bedrud-data:/var/lib/bedrud \
  ghcr.io/bedrud-ir/bedrud:latest
```

### Docker Compose

```yaml
services:
  bedrud:
    image: ghcr.io/bedrud-ir/bedrud:latest
    ports:
      - "8090:8090"
      - "7880:7880"
      - "50000-60000:50000-60000/udp"
    volumes:
      - bedrud-data:/var/lib/bedrud
    restart: unless-stopped

volumes:
  bedrud-data:
```

### Puertos

| Puerto | Servicio | Protocolo |
|--------|----------|-----------|
| 8090 | API + Interfaz Web | HTTP |
| 7880 | LiveKit | WebSocket + HTTP |
| 50000–60000 | Medios WebRTC | UDP |

### Configuración

Pasa un archivo de configuración o usa variables de entorno:

```bash
docker run -d --name bedrud \
  -p 8090:8090 \
  -p 7880:7880 \
  -e JWT_SECRET=change-this-to-a-random-string \
  -e LIVEKIT_API_KEY=prodkey \
  -e LIVEKIT_API_SECRET=prodsecret \
  -v bedrud-data:/var/lib/bedrud \
  ghcr.io/bedrud-ir/bedrud:latest
```

<Callout type="warning">
    Los secretos JWT y LiveKit predeterminados son solo para pruebas rápidas. Cámbialos antes de exponer el contenedor a internet.
</Callout>

Documentación completa de Docker incluyendo volúmenes, proxy inverso y configuración TLS: [Guía de Docker](/es/docs/guides/docker).

## 3. Instalación Interactiva

Si ya tienes el binario `bedrud` en el servidor:

```bash
sudo ./bedrud install
```

El instalador detecta automáticamente la IP de tu servidor y solicita:

| Prompt | Predeterminado | Notas |
|--------|----------------|-------|
| **Dirección IP** | Detectada automáticamente (primera IPv4 no loopback) | Presiona Enter para aceptar, o escribe una IP diferente |
| **Dominio** | Vacío (solo IP) | Si se proporciona con un correo, habilita TLS de Let's Encrypt |
| **Correo** | Vacío | Usado para registro de Let's Encrypt (solo si se configura dominio) |
| **TLS autofirmado** | Sí | Se ofrece cuando no se proporciona dominio. El predeterminado es `Y` |

Todas las solicitudes se omiten cuando stdin se canaliza (modo no interactivo): se usan indicadores y valores predeterminados.

## 4. No Interactivo (Basado en Indicadores)

Omite todas las solicitudes pasando indicadores.

### Let's Encrypt (se requiere dominio)

**Requiere:** Registro DNS A para tu dominio ya apuntando a la IP de este servidor.

```bash
sudo ./bedrud install --tls --domain meet.example.com --email webmaster@example.com
```

### TLS Autofirmado (basado en IP)

**Requiere:** Nada adicional. Funciona inmediatamente.

```bash
sudo ./bedrud install --tls --ip 1.2.3.4
```

### HTTP Simple (solo desarrollo / interno)

**Requiere:** localhost o `127.0.0.1`. Los navegadores bloquean cámara/micrófono en sitios no HTTPS excepto localhost.

```bash
sudo ./bedrud install --ip 1.2.3.4
```

### Autofirmado para un Dominio (pruebas previas a DNS)

**Requiere:** Nombre de dominio (DNS no necesita resolverse todavía).

```bash
sudo ./bedrud install --tls --domain meet.example.com
```

### Puerto Personalizado

**Requiere:** Puerto que no esté en uso.

```bash
sudo ./bedrud install --tls --port 8443 --domain meet.example.com
```

### Certificados Externos

**Requiere:** Archivos de cadena completa y clave privada en formato PEM.

```bash
sudo ./bedrud install --tls --cert /ruta/a/fullchain.pem --key /ruta/a/privkey.pem
```

### Puertos Personalizados de LiveKit

**Requiere:** Los tres puertos abiertos en tu firewall.

```bash
sudo ./bedrud install --lk-port 9000 --lk-tcp-port 9001 --lk-udp-port 9002
```

### Detrás de un Proxy Inverso

**Requiere:** Un proxy inverso (Nginx, Caddy, Traefik, Cloudflare) que termina TLS frente a Bedrud.

```bash
sudo ./bedrud install --behind-proxy --domain meet.example.com
```

Establece `behindProxy: true` en la configuración y configura encabezados de proxy de confianza.

### Servidor LiveKit Externo

**Requiere:** Un servidor LiveKit separado ya en ejecución, accesible desde esta máquina.

```bash
sudo ./bedrud install --external-livekit https://livekit.example.com
```

Omite la instalación de LiveKit integrado. La clave API y el secreto deben coincidir en ambas configuraciones.

### Dominio LiveKit Separado

**Requiere:** Un dominio que resuelva directamente a este servidor, omitiendo cualquier CDN.

```bash
sudo ./bedrud install --livekit-domain lk.example.com --domain meet.example.com
```

Útil cuando el dominio principal está detrás de Cloudflare pero los medios WebRTC necesitan una conexión directa.

### Referencia Completa de Indicadores

| Indicador | Predeterminado | Descripción |
|-----------|----------------|-------------|
| `--tls` | `false` | Habilitar HTTPS (autofirmado o Let's Encrypt) |
| `--self-signed` | `false` | Generar un certificado TLS autofirmado |
| `--no-tls` | `false` | Deshabilitar TLS completamente (anula `--tls`) |
| `--ip` | detectado automáticamente | Anular la dirección IP del servidor |
| `--domain` | `""` | Nombre de dominio para Let's Encrypt o certificado autofirmado |
| `--email` | `""` | Correo para registro de Let's Encrypt |
| `--port` | `443` (TLS) / `8090` (HTTP) | Anular el puerto de escucha |
| `--cert` | `""` | Ruta al archivo de certificado existente (PEM) |
| `--key` | `""` | Ruta al archivo de clave privada existente (PEM) |
| `--lk-port` | `7880` | Puerto API de LiveKit |
| `--lk-tcp-port` | `7881` | Puerto TCP RTC de LiveKit |
| `--lk-udp-port` | `7882` | Puerto UDP RTC de LiveKit |
| `--fresh` | `false` | Eliminar instalación existente antes de instalar |
| `--behind-proxy` | `false` | El servidor está detrás de un proxy inverso / CDN |
| `--external-livekit` | `""` | URL de un servidor LiveKit externo |
| `--livekit-domain` | `""` | Dominio separado para LiveKit (omita CDN) |

## 5. Totalmente Automatizado (CLI de Python)

Compila, carga y configura tu servidor desde tu máquina local.

**Requiere:** Python 3.12+, [uv](https://docs.astral.sh/uv/) (ejecutor de paquetes Python), acceso con clave SSH al servidor de destino, Go 1.24+ y Bun instalados localmente.

```bash
cd tools/cli
uv run python bedrud.py --auto-config \
  --ip <tu-ip-del-servidor> \
  --user root \
  --auth-key ~/.ssh/id_rsa \
  --domain meet.example.com \
  --acme-email admin@example.com
```

**Lo que hace:**

1. Compila el binario backend (`dist/bedrud`)
2. Lo comprime en `bedrud.tar.xz`
3. Lo carga en el servidor vía rsync
4. Limpia servicios en conflicto (Nginx, Apache, Caddy)
5. Configura el firewall (UFW)
6. Ejecuta la lógica interna de `install` en el servidor

| Indicador | Descripción |
|-----------|-------------|
| `--ip` | Dirección IP del servidor |
| `--user` | Usuario SSH (predeterminado: root) |
| `--auth-key` | Ruta a la clave privada SSH |
| `--domain` | Nombre de dominio para Let's Encrypt |
| `--acme-email` | Correo para Let's Encrypt |
| `--uninstall` | Eliminar Bedrud del servidor |

## Post-Instalación

<CreateAdmin />

### Verificar Servicios

```bash
systemctl status bedrud livekit
```

### Verificar la API

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:8090
```

Esperado: `200`

### Verificar el Firewall

```bash
sudo ufw status
```

Asegúrate de que los puertos `8090/tcp`, `7880/tcp` y `50000:60000/udp` estén permitidos:

```bash
sudo ufw allow 8090/tcp
sudo ufw allow 7880/tcp
sudo ufw allow 50000:60000/udp
```

### Ver Registros

```bash
tail -f /var/log/bedrud/bedrud.log
```

O vía journalctl:

```bash
journalctl -u bedrud -u livekit -f
```

## Actualización

### Gestor de Paquetes

```bash
# Ubuntu / Debian
sudo apt update && sudo apt upgrade bedrud

# Arch Linux
yay -Syu bedrud-bin

# Fedora / RHEL
sudo dnf upgrade bedrud
```

Los servicios se reinician automáticamente después de la actualización del paquete.

### Descarga de Binario

Descarga la última versión y vuelve a ejecutar la instalación. El instalador es seguro de volver a ejecutar: sobrescribe el binario y la configuración mientras preserva la base de datos:

```bash
curl -fsSL -o bedrud.tar.xz https://github.com/bedrud-ir/bedrud/releases/latest/download/bedrud_linux_amd64.tar.xz
tar -xJf bedrud.tar.xz && chmod +x bedrud
sudo ./bedrud install --fresh
```

<Callout type="tip">
    `--fresh` elimina primero la instalación anterior. Omítelo para mantener tu configuración y base de datos existentes.
</Callout>

### Docker

```bash
docker pull ghcr.io/bedrud-ir/bedrud:latest
docker stop bedrud && docker rm bedrud
docker run -d --name bedrud \
  -p 8090:8090 \
  -p 7880:7880 \
  -p 50000-60000:50000-60000/udp \
  -v bedrud-data:/var/lib/bedrud \
  ghcr.io/bedrud-ir/bedrud:latest
```

Con Docker Compose:

```bash
docker compose pull
docker compose up -d
```

## Desinstalación

Para eliminar completamente Bedrud:

```bash
sudo bedrud uninstall
```

Esto detiene los servicios, elimina las unidades systemd, borra el binario de `/usr/local/bin` y limpia `/etc/bedrud`, `/var/lib/bedrud` y `/var/log/bedrud`.

## Solución de Problemas

| Problema | Solución |
|----------|----------|
| **Puerto 80/443 en uso** | Detén servidores web en conflicto: `systemctl stop nginx apache2 caddy` |
| **Los servicios no inician** | Verifica registros: `journalctl -u bedrud -u livekit --no-pager -n 50` |
| **Problemas de certificado TLS** | Verifica DNS: `dig +short meet.example.com` - debe devolver la IP de este servidor |
| **Conflictos de puerto LiveKit** | Usa los indicadores `--lk-port`, `--lk-tcp-port`, `--lk-udp-port` para cambiar puertos |
| **Instalación exitosa pero 404 en interfaz web** | Interfaz no integrada. Compila con `make build` antes de `bedrud install`. Las compilaciones solo de binario omiten la interfaz |
| **Instalación exitosa pero servicios fallaron** | Busca fallos silenciosos: `journalctl -u bedrud --no-pager -n 100`. Verifica la configuración en `/etc/bedrud/config.yaml` |
| **Puerto ya en uso (proceso desconocido)** | Encuéntralo: `ss -tlnp \| grep 8090` o `ss -ulnp \| grep 7880` |
| **Firewall bloqueando medios** | Abre puertos: `sudo ufw allow 8090/tcp && sudo ufw allow 7880/tcp && sudo ufw allow 50000:60000/udp` |
| **Advertencias de certificado autofirmado** | Esperado. Consulta [Guía de TLS Interno](/es/docs/guides/internal-tls) para configuración de CA privada |
| **Sin video / WebRTC falló** | Se requiere HTTPS para acceso a cámara. Usa `--tls`. Asegúrate de que UDP 50000–60000 esté abierto |
| **Proxy inverso: IP de cliente incorrecta** | Usa el indicador `--behind-proxy`. Configura encabezados de proxy de confianza |
